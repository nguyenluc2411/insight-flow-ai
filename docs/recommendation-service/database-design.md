PostgreSQL Database Design
Recommendation Service
Insight Flow AI – Engagement Layer
1. Database Overview
1.1 Database Responsibilities

The recommendation-service database is responsible for storing and managing all recommendation-related business data generated by the recommendation engine.

The database supports:

recommendation generation tracking
recommendation lifecycle management
business rule management
recommendation audit history
operational analytics
recommendation metrics
1.2 Ownership Boundaries

The recommendation-service follows the Database Per Service microservice pattern.

This means the service exclusively owns and manages its own database schema.

Owned Business Data
Domain Area	Ownership
Recommendations	recommendation-service
Recommendation Rules	recommendation-service
Recommendation Audit Logs	recommendation-service
Recommendation Metrics	recommendation-service
1.3 Why Recommendation-Service Owns These Tables

The recommendation-service generates operational business intelligence and therefore must independently manage:

recommendation lifecycle
recommendation persistence
recommendation processing history
event processing traceability
business rule evaluation history

This ownership prevents tight coupling with:

inventory-service
ai-prediction-service
sales-service

The service stores only the data required for recommendation operations.

3.3 Foreign Key Strategy

Foreign keys are used selectively for internal consistency.

Example

3.4 Why Relationships Exist
recommendation_history

Tracks status lifecycle transitions.

recommendation_audit

Tracks event-processing operations and failures.

The design avoids excessive joins across unrelated domains to maintain microservice independence.

11.3 Architectural Rationale

The design intentionally separates:

operational recommendation data
audit tracking
historical lifecycle data
analytical metrics

This separation improves:

maintainability
scalability
query performance
operational clarity
Conclusion

The recommendation-service PostgreSQL schema is designed using enterprise-grade microservice database principles.

The design provides:

scalable recommendation persistence
lifecycle tracking
auditability
event traceability
operational flexibility
future scalability support

By combining normalized relational structures with selective JSONB flexibility, the schema achieves a balance between maintainability, performance, and adaptability suitable for modern Kafka-based event-driven systems.
